# 보고서 검증

`docs/reports/{slug}_report.md`를 읽고 품질 평가. **파일 수정하지 마.**
`docs/knowledge/{slug}.md`가 있으면 함께 읽어 일관성도 체크.

## Anti-Bias (자기 검증 편향 방지)
1. "이 주제를 처음 접하는 팀원이 이해할 수 있는가?"로 판단.
2. 비유가 느낌만 전달하고 메커니즘과 안 맞으면 감점.
3. 90점+ 느껴지면 단점 3개 더 찾은 후 재채점. 95점+는 출판 수준일 때만.
4. 핵심 수치/주장 최소 2개는 실제 출처와 대조. 불가하면 `[미대조]`.

---

## Step 1. 문서 로드
Frontmatter slug/date_created/sources 확인.

## Step 2. 하드 룰 (하나라도 위반 → 즉시 Fail)

| # | 하드 룰 |
|---|---|
| H-1 | References 없거나 URL/논문명+연도 출처 없음 |
| H-2 | Frontmatter slug/date_created/sources 누락 |
| H-3 | Deep Dive에 비유/예시 없이 용어 나열만 |
| H-4 | 수식 기반 논문인데 수식 구현 코드 없음 (서베이 면제) |
| H-5 | Pros & Cons에 단점 없음 |
| H-6 | Conclusion이 본문과 연결 안 되고 근거 없는 낙관/비관만 |

## Step 2.5. 팩트체크 (3개+ 핵심 주장)
```
✅ "{주장}" — 출처/knowledge 일치
⚠️ "{주장}" — 과장 또는 출처 약함
❌ "{주장}" — 사실 오류 또는 knowledge와 모순
```
❌ → [B] -5점, ⚠️ 3개+ → [D] -3점

## Step 3. 채점 (각 0~25, 총 100점)

**[A] 가독성 & 흐름** — 25: 배경→개념→장단점→총평 자연스러움 / 15: 전환 끊김 / 5: 사실 나열 / 0: 메모 수준
**[B] 설명 충실성** — 25: "왜"까지 설명 / 15: "무엇"만 / 5: 단순 요약 / 0: 없음
**[C] 비유 & 예시** — 25: 본질 정확 + 오해 없음 / 15: 일부 추상적 / 5: 1개뿐 / 0: 없음
**[D] 총평 실용성** — 25: 명확한 권고 + 근거 연결 / 15: 애매한 권고 / 5: 요약만 / 0: 없음

| 총점 | 등급 |
|---|---|
| 90+ | Pass | 70-89 | Needs Improvement | 50-69 | Conditional Fail | 0-49 | Fail |

## Step 4. 출력 형식
```
📄 [{slug}_report] 품질 검증 리포트
⛔ 하드 룰: H-1~H-6 [통과/위반]
🔬 팩트체크: {N}개 — ✅{N} ⚠️{N} ❌{N}
📊 [A]{점수} [B]{점수} [C]{점수} [D]{점수} = 총 {점수}점 → {등급}
⚠️ 불일치: (knowledge 대조 결과, 없으면 생략)
💡 보완: (섹션·문단 → 독자가 막히는 이유 → 수정 방법)
🧪 독자 테스트: Q1, Q2
🙋 한 줄 총평: (처음 읽는 사람 입장에서 가장 걸리는 점)
```
