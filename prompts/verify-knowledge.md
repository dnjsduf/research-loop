# 지식 DB 검증

`docs/knowledge/{slug}.md`를 읽고 품질 평가. **파일 수정하지 마.**

## Anti-Bias (자기 검증 편향 방지)
1. 모든 주장이 틀렸다고 가정 → 출처 확인. 없으면 자동 ⚠️.
2. "개발자가 코드를 짤 수 있는가?"로 판단.
3. 90점+ 느껴지면 재검토. 95점+는 흠잡을 데 없을 때만.
4. 출처 URL 가능하면 web_fetch로 대조. 불가하면 `[미대조]`.

---

## Step 1. 문서 로드
Frontmatter에서 `slug`, `date_created`, `sources` 확인.

## Step 2. 하드 룰 (하나라도 위반 → 즉시 Fail)

| # | 하드 룰 |
|---|---|
| H-1 | 포인터 항목에 Section/Figure/Table 단위 위치 표기가 없음 |
| H-2 | 출처 항목 없거나 URL/논문명+연도 형식 없음 |
| H-3 | Frontmatter slug/date_created/sources 누락 |
| H-4 | 코어 로직에 수도코드/Step-by-Step 없고 자연어만 |

## Step 2.5. 팩트체크 (5개+ 핵심 주장 추출)
각 주장별: 출처 존재? 출처 일치? 수치 정확?
```
✅ "{주장}" — 출처: {출처} — 일치
⚠️ "{주장}" — 출처 없음/불일치
❌ "{주장}" — 환각 의심
```
❌ 1개 → [A] -5점, ⚠️ 3개+ → -3점

## Step 3. 채점 (각 0~25, 총 100점)

**[A] 로직 구체성** — 25: 바로 코드 작성 가능 / 15: 일부 추상적 / 5: 전부 추상적 / 0: 산문체
**[B] 프로젝트 적합성** — 25: 파일명·모듈 수준 / 15: 레이어 종류만 / 5: 일반론 / 0: 없음
**[C] 트레이드오프** — 25: 수치·조건 포함 2개+ / 15: 추상적 경고 / 5: 장점 위주 / 0: 없음
**[D] 디테일 포인터** — 25: 3개+ Section/Figure 포인터 / 15: 1-2개 모호 / 5: 제목만 / 0: 없음

| 총점 | 등급 |
|---|---|
| 90+ | Pass | 70-89 | Needs Improvement | 50-69 | Conditional Fail | 0-49 | Fail |

## Step 4. 출력 형식
```
🔍 [{slug}] 품질 검증 리포트
⛔ 하드 룰: H-1~H-4 [통과/위반]
🔬 팩트체크: {N}개 검증 — ✅{N} ⚠️{N} ❌{N}
📊 [A]{점수} [B]{점수} [C]{점수} [D]{점수} = 총 {점수}점 → {등급}
💡 보완 제안: (낮은 항목부터, 원문 어느 섹션을 봐야 하는지)
🔥 압박 질문: Q1, Q2
```
